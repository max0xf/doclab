## Overview

### Purpose

This system provides a **format-agnostic framework** for rendering file content with **enrichments** - additional layers of information overlaid on the original content. The system supports:

- **Multiple file formats**: Markdown, YAML, XML, plain text, code
- **Multiple enrichment types**: PR diffs, user changes, comments, annotations
- **Two rendering modes**: Plain text (line-based) and Structured view (AST-based)
- **Inline editing**: Both plain text editing and rich text editing for structured formats

### Key Innovation: Virtual Content

The system uses **virtual content rendering** - a technique where the original file content is never modified directly. Instead:

1. **Original content** remains immutable
2. **Enrichments** are applied as independent layers
3. **Virtual (ephemeral) content** is generated by merging original + enrichments
4. **Rendering** displays the virtual content with visual indicators

This approach enables:
- Multiple enrichments to coexist without conflicts
- Independent application/removal of each enrichment
- Preservation of original content integrity
- Time-travel and version comparison

## Core Concepts

### 1. Enrichments

**Enrichments** are metadata layers attached to file content. They represent changes, annotations, or additional information without modifying the original file.

**Key Properties:**
- **Type**: Category of enrichment (diff, comment, annotation, highlight)
- **Category**: Grouping (diff, visual, metadata, annotation)
- **Priority**: Conflict resolution (higher priority wins)
- **Dual Mapping**: Maps to both raw file lines AND rendered blocks

**Enrichment Structure:**

```typescript
interface BaseEnrichment {
  id: string;
  type: 'comment' | 'diff' | 'annotation' | 'highlight' | 'warning' | 'error';
  
  // Dual mapping system
  rawMapping: RawMapping;        // Position in original file
  renderedMapping: RenderedMapping;  // Position in rendered output
  
  // Visual representation
  visual: EnrichmentVisual;
  
  // Metadata
  createdAt: string;
  createdBy: string;
  updatedAt?: string;
}
```

### 2. Dual Mapping System

The **dual mapping** system is critical for format-agnostic enrichments:

**Raw Mapping** (Source):
```typescript
interface RawMapping {
  startLine: number;      // Line in original file (1-indexed)
  endLine: number;
  startColumn?: number;
  endColumn?: number;
}
```

**Rendered Mapping** (Target):
```typescript
interface RenderedMapping {
  blockIds: string[];     // Which rendered blocks
  positions?: Array<{
    blockId: string;
    offset?: number;      // Character offset within block
    lineOffset?: number;  // Line offset within block
  }>;
}
```

**Why Dual Mapping?**
- **Raw mapping**: Anchors enrichment to source file (stable across renders)
- **Rendered mapping**: Positions enrichment in visual output (varies by format)
- **Bidirectional conversion**: EnrichmentMapper translates between them

### 3. Block Maps

**Block maps** connect raw file lines to rendered output:

```typescript
interface BlockMap {
  blockId: string;           // Unique identifier
  rawLines: number[];        // Which raw lines this block represents
  renderedLines?: number[];  // Which rendered lines (if applicable)
  type: string;              // Block type (e.g., 'code', 'yaml_mapping', 'md_heading')
  metadata?: Record<string, any>;
}
```

**Examples:**
- **Plain text**: One block per line (`plaintext-line-1`, `plaintext-line-2`, ...)
- **Markdown**: Blocks for headings, paragraphs, code blocks, lists
- **YAML**: Blocks for mappings, sequences, scalars
- **Code**: Blocks for functions, classes, statements

### 4. Virtual (Ephemeral) Content

**Virtual content** is the merged result of original content + all enrichments:

**Process:**
1. Start with original file content
2. Collect all diff enrichments (sorted by priority)
3. Apply each enrichment independently to original
4. Merge results, resolving conflicts by priority
5. Generate ephemeral lines with metadata

**Ephemeral Line Structure:**
```typescript
interface EphemeralLine {
  content: string;
  newLineNumber: number | null;  // null for deleted lines
  oldLineNumber: number | null;  // null for added lines
  changeType: 'add' | 'delete' | 'context' | null;
  enrichmentType: string;        // Which enrichment owns this line
}
```

## Enrichment System

### Enrichment Types

#### 1. Diff Enrichments

**Purpose**: Represent changes from Pull Requests or user edits

**Structure:**
```typescript
interface DiffEnrichment extends BaseEnrichment {
  type: 'diff';
  data: {
    changeType: 'added' | 'deleted' | 'modified';
    prNumber?: number;
    prTitle?: string;
    prAuthor?: string;
    oldContent?: string;
    newContent?: string;
  };
}
```

**Visual Representation:**
- **Added lines**: Green background, `+` marker
- **Deleted lines**: Red background, `-` marker, strikethrough
- **Modified lines**: Yellow background
- **Badge**: Shows PR number (clickable) or "Your Changes"

**Priority System:**
- PR diffs: Priority 1-5 (based on PR age/status)
- User changes: Priority 10 (highest - always visible)

**Conflict Resolution:**
When multiple diffs affect the same line:
1. Higher priority wins for display
2. Lower priority shown as "hidden" with indicator
3. User can toggle to see all versions

#### 2. Comment Enrichments

**Purpose**: Inline comments on specific lines or blocks

**Structure:**
```typescript
interface CommentEnrichment extends BaseEnrichment {
  type: 'comment';
  data: {
    text: string;
    resolved: boolean;
    replyCount: number;
    replies?: Array<{
      id: string;
      text: string;
      author: string;
      createdAt: string;
    }>;
  };
}
```

**Visual Representation:**
- **Marker**: Orange badge with comment icon
- **Tooltip**: Comment preview
- **Click**: Opens comment panel
- **Indicator**: Shows reply count

#### 3. Annotation Enrichments

**Purpose**: Warnings, errors, notes, suggestions

**Structure:**
```typescript
interface AnnotationEnrichment extends BaseEnrichment {
  type: 'annotation';
  data: {
    title: string;
    message: string;
    severity: 'info' | 'warning' | 'error';
    category?: string;
  };
}
```

**Visual Representation:**
- **Info**: Blue underline
- **Warning**: Yellow underline, warning icon
- **Error**: Red underline, error icon

#### 4. Highlight Enrichments

**Purpose**: Search results, references, selections

**Structure:**
```typescript
interface HighlightEnrichment extends BaseEnrichment {
  type: 'highlight';
  data: {
    reason: string;
    category?: string;
  };
}
```

**Visual Representation:**
- **Background**: Yellow highlight
- **Tooltip**: Highlight reason

### Enrichment Lifecycle

```
1. Creation
   ↓
2. Conversion to Unified Format
   • fileCommentToEnrichment()
   • prDiffToEnrichments()
   ↓
3. Dual Mapping Generation
   • EnrichmentMapper.ensureDualMapping()
   ↓
4. Grouping & Filtering
   • groupByBlock()
   • groupByLine()
   • filterByType()
   ↓
5. Priority Sorting
   • sortByPriority()
   ↓
6. Rendering
   • Applied to blocks/lines
   • Visual indicators added
   ↓
7. Interaction
   • Click handlers
   • Tooltips
   • Actions (resolve, reply, etc.)
```

### Enrichment Mapper

The **EnrichmentMapper** class handles all enrichment-to-block mapping:

**Key Methods:**

```typescript
class EnrichmentMapper {
  // Convert raw line position to rendered blocks
  rawToRendered(rawMapping: RawMapping): RenderedMapping;
  
  // Convert rendered blocks back to raw lines
  renderedToRaw(renderedMapping: RenderedMapping): RawMapping | null;
  
  // Ensure enrichment has both mappings
  ensureDualMapping(enrichment: Enrichment): Enrichment;
  
  // Group enrichments by block ID
  groupByBlock(enrichments: Enrichment[]): Map<string, Enrichment[]>;
  
  // Group enrichments by raw line
  groupByLine(enrichments: Enrichment[]): Map<number, Enrichment[]>;
  
  // Filter enrichments by type
  filterByType<T>(enrichments: Enrichment[], type: T): Enrichment[];
  
  // Sort enrichments by priority
  sortByPriority(enrichments: Enrichment[]): Enrichment[];
}
```

## Rendering Pipeline

### Phase 1: Content Type Detection

**Input**: Filename + file content

**Process:**
1. Check file extension against known types
2. If unknown, analyze content patterns
3. Determine content type and language

**Output**: `ContentTypeInfo`

```typescript
interface ContentTypeInfo {
  type: 'markdown' | 'code' | 'yaml' | 'xml' | 'plaintext';
  language?: string;  // For code: 'javascript', 'python', etc.
  extension?: string;
}
```

**Detection Rules:**
- **Markdown**: `.md`, `.markdown` extensions OR markdown patterns (headings, lists, links)
- **YAML**: `.yaml`, `.yml` extensions
- **XML**: `.xml` extension OR `<?xml` declaration
- **Code**: Known extensions (`.js`, `.py`, etc.) OR code patterns (imports, functions)
- **Plain text**: Default fallback

### Phase 2: Renderer Selection

**Input**: `ContentTypeInfo`

**Process:**
1. Query `FormatRendererRegistry`
2. Each renderer checks `canHandle(contentType)`
3. Return first matching renderer

**Output**: `FormatRenderer` instance

**Renderer Priority:**
1. MarkdownFormatRenderer (for markdown)
2. YAMLFormatRenderer (for YAML)
3. CodeFormatRenderer (for code)
4. PlainTextRenderer (fallback)

### Phase 3: Parsing

**Input**: File content + enrichments

**Process:**
1. Build virtual content (merge all diff enrichments)
2. Parse content to AST (if structured format)
3. Generate block maps
4. Extract metadata

**Output**: `ParseResult`

```typescript
interface ParseResult {
  ast: any;              // Format-specific AST
  blocks: BlockMap[];    // Block ID to line mapping
  metadata?: Record<string, any>;
}
```

**Virtual Content Building (for diffs):**

```
1. Start with original file lines
2. For each diff enrichment (sorted by priority):
   a. Extract add/delete operations
   b. Track position in original file
   c. Insert deletions (marked for strikethrough)
   d. Insert additions (marked as new)
3. Merge all changes into ephemeral lines
4. Assign line numbers:
   - newLineNumber: Position in virtual content
   - oldLineNumber: Position in original file
5. Tag each line with enrichmentType
```

### Phase 4: Enrichment Mapping

**Input**: `ParseResult` + enrichments

**Process:**
1. Create `EnrichmentMapper` with block maps
2. Convert enrichments to dual mappings
3. Group enrichments by block/line
4. Sort by priority

**Output**: Mapped enrichments ready for rendering

### Phase 5: Rendering

**Input**: `ParseResult` + mapped enrichments + render options

**Process:**
1. Iterate through blocks/lines
2. For each block:
   a. Get enrichments for this block
   b. Determine visual styling
   c. Add markers/badges
   d. Attach interaction handlers
3. Generate React components

**Output**: Rendered React tree

## Format Renderers

### Base Interface

All format renderers implement this interface:

```typescript
interface FormatRenderer {
  name: string;
  supportedTypes: ContentTypeInfo['type'][];
  supportedEnrichments: Enrichment['type'][];
  
  parse(content: string, contentType: ContentTypeInfo, prDiffData?: any): ParseResult;
  render(parseResult: ParseResult, enrichments: Enrichment[], options: RenderOptions): ReactNode;
  canHandle(contentType: ContentTypeInfo): boolean;
}
```

### PlainTextRenderer

**Purpose**: Line-based rendering for plain text and fallback

**Parsing:**
- Splits content into lines
- Creates one block per line
- Builds virtual content with diff enrichments

**Rendering:**
- Renders each line with `LineRenderer` component
- Shows line numbers (old/new)
- Applies color coding (green/red for add/delete)
- Displays badges for enrichment type
- Groups contiguous changes into navigation chunks

**Block Structure:**
```typescript
{
  blockId: 'plaintext-line-1',
  rawLines: [1],
  type: 'plaintext_line'
}
```

**Features:**
- Chunk navigation (prev/next change)
- Inline editing (contentEditable)
- Click to comment
- Badge display (PR number, "Your Changes")

### MarkdownFormatRenderer

**Purpose**: Structured rendering for Markdown files

**Parsing:**
1. Build virtual content (with deleted lines from diffs)
2. Parse with `unified` + `remark-parse` + `remark-gfm`
3. Convert MDAST to UnifiedNode format
4. Track line positions for each node
5. Create blocks for structural elements

**Rendering:**
- Uses `MarkdownTreeView` component
- Renders as collapsible tree structure
- Preserves Markdown semantics
- Shows diff indicators on blocks

**Block Structure:**
```typescript
{
  blockId: 'md-heading-1',
  rawLines: [1, 2],
  type: 'heading',
  metadata: { level: 1, text: 'Introduction' }
}
```

**Node Types:**
- Headings (h1-h6)
- Paragraphs
- Lists (ordered/unordered)
- Code blocks
- Blockquotes
- Tables
- Links, images

**Diff Handling:**
- Deleted blocks: Shown with strikethrough, red background
- Added blocks: Green background
- Modified blocks: Yellow background
- Nested diffs: Propagate to children

### YAMLFormatRenderer

**Purpose**: Structured rendering for YAML files

**Parsing:**
1. Build virtual content (with deleted lines)
2. Parse with `js-yaml`
3. Convert to tree structure
4. Map YAML paths to line numbers
5. Create blocks for mappings/sequences

**Rendering:**
- Uses `YAMLTreeView` component
- Collapsible tree with indentation
- Shows keys, values, types
- Diff indicators on nodes

**Block Structure:**
```typescript
{
  blockId: 'yaml-mapping-config.database',
  rawLines: [10, 11, 12],
  type: 'yaml_mapping',
  metadata: { path: 'config.database', keys: ['host', 'port'] }
}
```

**Node Types:**
- Mappings (key-value pairs)
- Sequences (arrays)
- Scalars (strings, numbers, booleans)

**Diff Handling:**
- Structural comparison (deep diff)
- Shows added/deleted/modified keys
- Highlights value changes
- Preserves YAML structure

### CodeFormatRenderer

**Purpose**: Syntax-highlighted code rendering

**Parsing:**
1. Detect language from extension/content
2. Parse with language-specific parser (if available)
3. Create blocks for code structures
4. Apply syntax highlighting

**Rendering:**
- Uses `rehype-expressive-code`
- Syntax highlighting with themes
- Line numbers
- Code block features (copy, expand)

**Block Structure:**
```typescript
{
  blockId: 'code-function-main',
  rawLines: [15, 16, 17, 18, 19],
  type: 'function',
  metadata: { name: 'main', language: 'python' }
}
```

**Features:**
- Syntax highlighting
- Line numbers
- Copy button
- Expand/collapse functions
- Diff highlighting

## Rendering Modes

### Mode 1: Plain Text Mode

**Characteristics:**
- Line-based rendering
- Always available (fallback)
- Direct line number mapping
- Simple block structure

**Use Cases:**
- Plain text files
- Unknown file types
- Large files (performance)
- Simple diffs

**Renderer**: `PlainTextRenderer`

**Visual Elements:**
- Line numbers (old/new)
- Line content
- Background colors (add/delete)
- Badges (PR, user changes)
- Navigation buttons

### Mode 2: Structured View Mode

**Characteristics:**
- AST-based rendering
- Format-specific structure
- Semantic blocks
- Collapsible tree

**Use Cases:**
- Markdown files
- YAML configuration
- XML documents
- Structured data

**Renderers**: 
- `MarkdownFormatRenderer`
- `YAMLFormatRenderer`
- (Future: `XMLFormatRenderer`)

**Visual Elements:**
- Tree structure
- Collapse/expand controls
- Semantic styling
- Block-level diffs
- Nested enrichments

### Mode Switching

Users can switch between modes:

```
Plain Text ←→ Structured View
```

**Persistence:**
- Mode preference saved per file type
- Enrichments work in both modes
- Line mappings preserved

## Edit Modes

### Edit Mode in Plain Text

**Mechanism**: `contentEditable` on line elements

**Process:**
1. User clicks "Edit" button
2. Lines become editable
3. User modifies content
4. On blur, capture changes
5. Generate diff hunks
6. Save as user changes enrichment

**Features:**
- Inline editing
- Real-time preview
- Undo/redo
- Save/discard

**Implementation:**
```typescript
<code
  contentEditable={isEditable && changeType !== 'delete'}
  suppressContentEditableWarning
  onBlur={e => {
    if (isEditable && onLineEdit) {
      onLineEdit(lineNumber - 1, e.currentTarget.textContent || '');
    }
  }}
>
  {lineContent}
</code>
```

### Edit Mode in Structured View

**Mechanism**: Rich text editing with structure preservation

**Approach 1: Markdown Editor**
- Use markdown-aware editor (e.g., MDX Editor, Toast UI)
- Parse input to AST
- Preserve structure
- Generate diff from AST changes

**Approach 2: WYSIWYG Editor**
- Visual editing of rendered content
- Track changes at block level
- Convert back to source format
- Generate structural diff

**Challenges:**
- Maintaining format validity
- Handling complex structures
- Preserving enrichments during edit
- Conflict resolution

**Recommended Implementation:**
```typescript
// For Markdown
<MarkdownEditor
  value={content}
  onChange={newContent => {
    const diff = generateMarkdownDiff(content, newContent);
    saveUserChanges(diff);
  }}
  preserveStructure={true}
  showPreview={true}
/>
```

### User Changes Workflow

```
1. User enters edit mode
   ↓
2. Makes changes
   ↓
3. System generates diff
   • Compare original vs modified
   • Create unified diff hunks
   • Track line numbers
   ↓
4. Save as user changes enrichment
   • Type: 'user-changes'
   • Category: 'diff'
   • Priority: 10 (highest)
   ↓
5. Render with other enrichments
   • Blue badge: "Your Changes"
   • Distinct from PR diffs
   ↓
6. User can:
   • Continue editing
   • Discard changes
   • Commit to PR
```

## Implementation Guide

### Technology Stack Requirements

**Frontend:**
- React or equivalent component framework
- State management (React hooks, Redux, MobX, etc.)
- Markdown parser: `unified` + `remark-parse` + `remark-gfm`
- YAML parser: `js-yaml` or equivalent
- XML parser: `fast-xml-parser` or equivalent
- Syntax highlighting: `rehype-highlight` or `prism`

**Backend:**
- REST API for enrichments (comments, diffs, user changes)
- Git integration for PR diffs
- Database for storing enrichments
- Diff generation library

### Implementation Steps

#### 1. Core Types

```typescript
// Define all core types
```

## Supported File Formats

### Markdown

**Extensions**: `.md`, `.markdown`, `.mdown`, `.mkd`

**Parser**: `unified` + `remark-parse` + `remark-gfm`

**AST**: MDAST (Markdown Abstract Syntax Tree)

**Blocks**:
- Headings (h1-h6)
- Paragraphs
- Lists (ordered, unordered, task lists)
- Code blocks (fenced, indented)
- Blockquotes
- Tables
- Horizontal rules
- Links, images

**Enrichments**:
- Comments on blocks
- Diffs on blocks (added/deleted/modified)
- Annotations on specific elements

**Rendering**:
- Collapsible tree structure
- Syntax highlighting in code blocks
- GitHub Flavored Markdown support
- Preview mode

### YAML

**Extensions**: `.yaml`, `.yml`

**Parser**: `js-yaml`

**AST**: YAML document tree

**Blocks**:
- Mappings (key-value pairs)
- Sequences (arrays)
- Scalars (strings, numbers, booleans, null)
- Nested structures

**Enrichments**:
- Comments on keys/values
- Diffs on structural changes
- Validation errors/warnings

**Rendering**:
- Collapsible tree with indentation
- Type indicators (mapping, sequence, scalar)
- Value preview
- Path display

### XML

**Extensions**: `.xml`, `.xsd`, `.xsl`, `.xslt`

**Parser**: `fast-xml-parser` or DOM parser

**AST**: XML DOM tree

**Blocks**:
- Elements
- Attributes
- Text nodes
- CDATA sections
- Comments

**Enrichments**:
- Comments on elements
- Diffs on structure
- Schema validation errors

**Rendering**:
- Collapsible tree
- Attribute display
- Namespace handling
- Syntax highlighting

### Plain Text / Code

**Extensions**: All others

**Parser**: Line-based splitting (plain text) or language-specific (code)

**AST**: Line array or language AST

**Blocks**:
- Lines (plain text)
- Functions, classes, statements (code)

**Enrichments**:
- Comments on lines
- Diffs on lines
- Linter warnings/errors

**Rendering**:
- Line-by-line display
- Syntax highlighting (code)
- Line numbers
- Diff indicators

## Summary

This virtual content rendering system provides a **flexible, extensible framework** for displaying file content with enrichments across multiple formats. Key strengths:

1. **Format-agnostic**: Works with any file format via renderer plugins
2. **Non-destructive**: Original content never modified
3. **Multi-layered**: Multiple enrichments coexist independently
4. **Dual mapping**: Enrichments work in both raw and rendered views
5. **Extensible**: Easy to add new enrichment types and formats
6. **Interactive**: Full editing support with inline changes
7. **Performant**: Optimized for large files and many enrichments

The system is production-ready and can be implemented in any modern web stack with appropriate parser libraries.
